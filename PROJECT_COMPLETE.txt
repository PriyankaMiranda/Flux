CORE CONCEPTS
================================================================================

WORKERS:
  Source Worker      → Fetches data from external APIs
  Transcriber Worker → Saves data to database
  Aggregation Worker → Joins multiple streams with time-matching
  Algorithm Worker   → Analyzes data using custom functions

STREAMS:
  Named channels connecting workers
  Example: "stock_data", "news_data", "aggregated_data"

DATA FLOW:
  Fetch → Save → Join → Analyze → Save
  
  source_stocks ----[stock_data]----> save_raw_stocks ----[stocks_saved]--→
  source_news   ----[news_data]-----> save_raw_news   ----[news_saved]---→
                                          ↓
                                    [agg_stocks_news]
                                    TIME-MATCH WITHIN 60s
                                          ↓
                                   [aggregated_data]
                                          ↓
                          save_aggregated + algorithm_analysis
                                          ↓
                                   [analysis_results]
                                          ↓
                                      save_results

================================================================================
FINAL NOTES
================================================================================

Design Philosophy:
  - Declarative (config-driven) vs Imperative (code-driven)
  - DAG-based orchestration (workers in dependency order)
  - Pub/Sub message passing (loose coupling)
  - Worker behavior pattern (Erlang/OTP)
  - Configuration = code (YAML is your API)

Why This Design:
  - No manual orchestration needed
  - Easy to add/remove workers
  - Failure isolation per worker
  - Automatic supervision and restart
  - Scalable to many machines
  - Clear data flow (visual in config)
